# simple-android-info

[English](README.en.md)

Программа для устройств на Android: телефонов, телевизоров и приставок. Она подключается через кабель, собирает сведения об устройстве и проверяет, какие основные узлы системы реально работают после включения или прошивки. Права суперпользователя не нужны.

Готовые исполняемые файлы лежат в каталоге [`bin/`](bin/). Для обычной работы собирать проект из исходников не требуется.

Перед запуском нужно узнать архитектуру процессора устройства:

```powershell
adb shell getprop ro.product.cpu.abi
```

| Файл | Архитектура | Для каких устройств |
|------|-------------|---------------------|
| [`bin/simple-android-info-aarch64`](bin/simple-android-info-aarch64) | 64-разрядный ARM (`arm64-v8a`) | большинство современных телефонов и телевизоров |
| [`bin/simple-android-info-armeabi-v7a`](bin/simple-android-info-armeabi-v7a) | 32-разрядный ARM (`armeabi-v7a`) | старые устройства и часть приставок |

Если архитектура не совпадёт с файлом, система может ответить «файл не найден», хотя файл на устройстве есть.

## Как запустить

Понадобятся программа Android Debug Bridge (`adb`) и включённая отладка по USB на устройстве.

Файл нужно копировать в каталог `/data/local/tmp/`. На карту памяти и в общую память (`/sdcard`, `/storage`) программа обычно не запускается: там запрещено выполнение файлов.

```powershell
# для 64-разрядного ARM
adb push bin\simple-android-info-aarch64 /data/local/tmp/simple-android-info

# для 32-разрядного ARM
# adb push bin\simple-android-info-armeabi-v7a /data/local/tmp/simple-android-info

adb shell chmod 755 /data/local/tmp/simple-android-info
adb shell /data/local/tmp/simple-android-info
```

Если после копирования система пишет, что нет прав на выполнение, снова выполните команду `chmod`, как выше.

### Параметры запуска

| Параметр | Что делает |
|----------|------------|
| без параметров | полный отчёт: по каждому пункту проверки выводятся факты |
| `--lang=ru` или `--lang=en` | то же, плюс пояснения на русском или английском (что проверяется, откуда берутся данные, зачем это нужно). То же самое: `--explain=ru` или `--explain=en` |
| `--summary` или `-s` | только короткая сводная таблица, без списка проверок |
| `--smt` | то же, что запуск без параметров |
| `--class=phone`, `--class=tv` или `--class=box` | принудительно считать устройство телефоном, телевизором или приставкой |
| `--class=summary` | то же, что `--summary` |
| `--json` или `-j` | результат в формате JSON |
| `-v` или `--verbose` | дополнительно вывести список устройств на шинах (I2C, SPI, сеть и другие) |

Примеры:

```powershell
adb shell /data/local/tmp/simple-android-info
adb shell /data/local/tmp/simple-android-info --lang=ru
adb shell /data/local/tmp/simple-android-info --lang=en
adb shell /data/local/tmp/simple-android-info --summary
adb shell /data/local/tmp/simple-android-info --json --lang=ru
adb shell /data/local/tmp/simple-android-info --class=tv
```

### Код завершения

| Код | Значение |
|-----|----------|
| `0` | обязательных ошибок нет (режим `--summary` всегда завершается с кодом `0`) |
| `1` | хотя бы одна обязательная проверка не прошла |
| `2` | неверный параметр командной строки |

## Как читать результат

В обычном режиме сначала печатается краткая шапка: производитель, модель, серийный номер, версия Android. Дальше идут отдельные блоки проверок.

Каждый блок выглядит так:

```text
[PASS] имя_проверки
  строки с фактами

  ———
  что:    …
  где:    …
  зачем:  …
```

Три строки «что / где / зачем» появляются только при `--lang=ru` (на английском при `--lang=en` это `what` / `where` / `why`).

В конце печатается итог: `RESULT: PASS` или `RESULT: FAIL`.

### Статусы проверок

| Статус | Значение |
|--------|----------|
| `PASS` | узел найден и выглядит работоспособным |
| `FAIL` | для этого типа устройства узел ожидался, но не найден или не готов; программа завершится с кодом `1` |
| `SKIP` | для этого типа устройства проверка не обязательна |

### Список проверок

| Имя | Что проверяется |
|-----|-----------------|
| `boot_completed` | система Android полностью загрузилась и готова к работе программ |
| `verified_boot` | проверка целостности прошивки при загрузке (состояние «зелёный», «жёлтый» и другие) |
| `nvram` | заводская область настроек и радиокалибровок инициализирована (типично для чипов MediaTek и похожих платформ) |
| `storage` | флеш-память (встроенный накопитель) видна операционной системе |
| `data_mounted` | раздел пользовательских данных `/data` смонтирован и известен его объём |
| `display` | работает вывод изображения: есть разрешение экрана и служба отрисовки |
| `gpu` | обнаружен графический ускоритель |
| `audio` | есть звуковая карта и служба звука |
| `wlan` | есть сетевой интерфейс Wi‑Fi и поднята служба Wi‑Fi |
| `ethernet` | есть проводная сеть (обычно для телевизора или приставки, если нет Wi‑Fi) |
| `bluetooth` | система видит контроллер Bluetooth |
| `touch` | зарегистрирован сенсорный экран (для телефонов и планшетов) |
| `battery_charger` | видны батарея, зарядка или питание по USB (для телефонов) |
| `camera` | служба камер сообщает хотя бы одну камеру |
| `sensors` | служба датчиков или шина датчиков показывает устройства (акселерометр, гироскоп и другие) |
| `modem_baseband` | загружена прошивка модема и есть сетевые интерфейсы модема |
| `gnss` | подсистема спутниковой навигации (GPS и аналоги) доступна |
| `hdmi` | есть внешний видеовыход HDMI или живой графический вывод (для телевизора и приставки) |
| `usb` | устройство видно компьютеру по USB (отладка, передача файлов) |
| `surfaceflinger` | запущена служба, которая собирает изображение на экране |
| `identity` | сводка об устройстве: серийный номер, платформа, процессор, память, время работы |

Заводской адрес Wi‑Fi на новых Android часто закрыт без прав суперпользователя. В сводке поле адреса сети в таком случае будет `n/a` («нет данных») — это ожидаемо.

### Сводная таблица (`--summary`)

| Поле | Содержание |
|------|------------|
| `class` | тип устройства: телефон, телевизор, приставка или неизвестно |
| `product`, `device`, `serial`, `hardware` | производитель, модель, внутреннее имя, серийный номер, обозначение платы |
| `android`, `build`, `boot` | версия системы, идентификатор сборки, слот загрузки и состояние проверки прошивки |
| `soc`, `platform`, `cpu`, `cpu_model`, `cpu_freq`, `ram` | чипсет, платформа, число ядер, модель процессора, частоты, оперативная память |
| `gpu`, `display`, `refresh` | графика, разрешение экрана, частота обновления |
| `storage`, `data` | тип накопителя и занятость раздела `/data` |
| `network`, `wlan_mac` | какие сети есть; адрес Wi‑Fi, если система его отдаёт |
| `timezone`, `uptime` | часовой пояс, время работы и средняя нагрузка |
| `buses`, `features` | число устройств на шинах и найденные возможности (тач, камера и другие) |

### Формат JSON

```powershell
adb shell /data/local/tmp/simple-android-info --json --lang=ru > report.json
adb shell /data/local/tmp/simple-android-info --summary --json > summary.json
```

В полном отчёте есть поля: режим работы, сводка, список проверок и общий результат (`PASS` или `FAIL`). При `-v` добавляется инвентарь шин. У каждой проверки есть имя, статус и текст с фактами; при `--lang=` добавляется блок пояснений.

### Как определяется тип устройства

1. По признакам телевизора → тип «телевизор».
2. Есть модем или признаки телефонии → тип «телефон».
3. Модема нет, но есть сеть или HDMI → тип «приставка».
4. Иначе → «неизвестно». В этом случае набор проверок всё равно подстраивается по фактическому железу (например, по модему или батарее).

## Сборка из исходников

Нужны: компилятор Rust, комплект Android NDK и программа `adb`.

Установите цели сборки:

```powershell
rustup target add aarch64-linux-android armv7-linux-androideabi
```

В файле [`.cargo/config.toml`](.cargo/config.toml) укажите пути к компоновщику из вашего NDK (подставьте свой каталог установки):

```toml
[target.aarch64-linux-android]
linker = "C:\\path\\to\\ndk\\toolchains\\llvm\\prebuilt\\windows-x86_64\\bin\\aarch64-linux-android24-clang.cmd"
ar     = "C:\\path\\to\\ndk\\toolchains\\llvm\\prebuilt\\windows-x86_64\\bin\\llvm-ar.exe"

[target.armv7-linux-androideabi]
linker = "C:\\path\\to\\ndk\\toolchains\\llvm\\prebuilt\\windows-x86_64\\bin\\armv7a-linux-androideabi24-clang.cmd"
ar     = "C:\\path\\to\\ndk\\toolchains\\llvm\\prebuilt\\windows-x86_64\\bin\\llvm-ar.exe"
```

Сборка и копирование в `bin/`:

```powershell
cargo build --release --target aarch64-linux-android
cargo build --release --target armv7-linux-androideabi
New-Item -ItemType Directory -Force bin | Out-Null
Copy-Item -Force target\aarch64-linux-android\release\simple-android-info bin\simple-android-info-aarch64
Copy-Item -Force target\armv7-linux-androideabi\release\simple-android-info bin\simple-android-info-armeabi-v7a
```

Каталог промежуточной сборки `target/` в репозиторий не кладётся. В git остаются исходники и готовые файлы в `bin/`.

## Ограничения

- Устройство должно быть уже загружено в Android, а кабель с отладкой — подключён.
- Без прав суперпользователя часть сведений недоступна (часто адрес Wi‑Fi и частота графического процессора).
- Имена узлов в системе зависят от производителя платы. Для распространённых платформ MediaTek, Amlogic и похожих типичные пути учтены.

Справка по параметрам на самом устройстве:

```text
simple-android-info --help
```
